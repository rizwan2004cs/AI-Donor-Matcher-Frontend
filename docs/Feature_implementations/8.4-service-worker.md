# Feature 8.4 — Service Worker

> **Branch:** `feat/service-worker`
> **Files:** `public/service-worker.js`, registration in `src/main.jsx`

---

## Overview

A service worker that enables offline support. Uses a **cache-first** strategy for static assets (HTML, CSS, JS, images) and a **network-first** strategy for API calls, so the app shell loads instantly even without a connection.

## Service Worker

```js
const CACHE_NAME = "donor-matcher-v1";

// Static assets to pre-cache on install
const PRECACHE_URLS = [
  "/",
  "/index.html",
  "/manifest.json",
  "/icon-192.png",
  "/icon-512.png",
];

// Install — pre-cache shell
self.addEventListener("install", (e) => {
  e.waitUntil(
    caches.open(CACHE_NAME).then((cache) => cache.addAll(PRECACHE_URLS))
  );
  self.skipWaiting();
});

// Activate — clean old caches
self.addEventListener("activate", (e) => {
  e.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(keys.filter((k) => k !== CACHE_NAME).map((k) => caches.delete(k)))
    )
  );
  self.clients.claim();
});

// Fetch — strategy router
self.addEventListener("fetch", (e) => {
  const { request } = e;

  // API calls → network-first
  if (request.url.includes("/api/")) {
    e.respondWith(
      fetch(request)
        .then((res) => {
          const clone = res.clone();
          caches.open(CACHE_NAME).then((cache) => cache.put(request, clone));
          return res;
        })
        .catch(() => caches.match(request))
    );
    return;
  }

  // Static assets → cache-first
  e.respondWith(
    caches.match(request).then((cached) => cached || fetch(request))
  );
});
```

## Registration in main.jsx

Add at the bottom of `src/main.jsx`, **after** `ReactDOM.createRoot(...).render(...)`:

```js
// Register service worker
if ("serviceWorker" in navigator) {
  window.addEventListener("load", () => {
    navigator.serviceWorker
      .register("/service-worker.js")
      .then((reg) => console.log("SW registered:", reg.scope))
      .catch((err) => console.error("SW registration failed:", err));
  });
}
```

## Caching Strategies

| Request Type | Strategy | Behaviour |
|-------------|----------|-----------|
| Static assets (HTML, CSS, JS, images) | Cache-first | Serve from cache instantly; fall back to network if not cached |
| API calls (`/api/*`) | Network-first | Try network; on failure, serve stale cached response |
| Map tiles (external CDN) | Not cached | Rely on browser HTTP cache / Leaflet tile cache |

---

## Cache Versioning

When deploying a new version:

1. Bump `CACHE_NAME` → `"donor-matcher-v2"`
2. The `activate` event deletes old caches automatically
3. `self.skipWaiting()` + `self.clients.claim()` ensures the new SW takes over immediately

---

## Behaviour Notes

- `self.skipWaiting()` activates the new service worker immediately (no waiting for tabs to close).
- `self.clients.claim()` lets the new SW control already-open pages.
- API responses are cloned before caching because a Response body can only be consumed once.
- In development, the SW may interfere with hot-module replacement. Unregister it in DevTools → Application → Service Workers if needed.
